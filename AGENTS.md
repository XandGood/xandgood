**权衡说明：** 下列准则优先考虑谨慎性而非速度。对于简单琐碎任务，请自行判断。

## 1. 编码前先思考

**不做假设。不隐瞒困惑。明确权衡取舍。**

在实施之前：
- 明确陈述你的假设。如果不确定，就提问。
- 如果存在多种解读方式，全部列出——不要默不作声地自行选择。
- 如果存在更简单的方案，主动说明。必要时提出反驳。
- 若有任何不明确之处，立即停止。指出具体困惑点并提问。
- 如果一项任务需要删除超过 3 个文件，请先停下来获得确认再继续。

## 2. 简单至上

**编写能解决问题的最少代码。不添加任何推测性内容。**

- 不实现需求之外的任何功能。
- 不为只使用一次的代码创建抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 如果你写了 200 行代码但其实 50 行就能解决，重写它。
- 如果你认为在某个地方的复杂是必要的，请先明确指出复杂和简单之间的不同，交由用户决策。

问问自己：“资深工程师会说这过于复杂吗？” 如果答案是肯定的，就简化。

## 3. 精准修改

**只修改必须修改的部分。只清理你自己造成的混乱。**

编辑现有代码时：
- 不要“优化”相邻的代码、注释或格式。
- 不要重构没有问题的代码。
- 遵循现有代码风格，即使你有不同的偏好。
- 如果你发现无关的死代码，提出来——但不要直接删除。
- 当需要完全重新修改或者切换其他方案时，注意先恢复原状，再进行改动。

当你的修改产生无用代码时：
- 删除因你的修改而变得无用的导入、变量或函数并及时告知。
- 除非被要求，否则不要删除原本就存在的死代码。

检验标准：每一处改动行都应能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到验证通过。需要真实的测试通过而非强行拟合测试结果，如果多次修改始终无法通过则应当停止验证、分析其原因并告知用户**

将任务转化为可验证的目标：
- “添加验证” → “编写针对无效输入的测试，然后令其通过”
- “修复 bug” → “编写一个可复现该 bug 的测试，然后令其通过”
- “重构 X” → “确保重构前后所有测试都能通过”

对于多步骤任务，给出一个简明的计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

明确的成功标准能让你独立完成迭代。模糊的标准（如 “让它能运行”）则需要不断澄清。
